🧭 How I Learn (My Methods)
I don't believe in passive learning. Tutorials and certifications are starting points — not proof of skill. Here's the loop I run on every new topic:
1. Read the official docs first, blog posts second
Most courses simplify and skip edge cases. Official docs (Playwright, Selenium, Postman, Anthropic API) are the source of truth. I read them end-to-end before touching any tutorial.
2. Build a small, broken thing
I never learn a tool by following along. I pick a tiny project — like the SauceDemo framework — and force myself to make decisions: which pattern, which folder structure, which assertion library. Mistakes teach more than copy-paste.
3. Break it deliberately
Once it works, I break it. Wrong selectors, race conditions, expired auth tokens, network throttling. The goal is to see how the tool behaves under stress, not just on the happy path.
4. Write about it
If I can't explain a concept clearly in writing, I haven't actually learned it. I keep a private learning journal and turn the strongest entries into LinkedIn posts.
5. Teach it forward
Mentoring junior QAs at Tandem forces me to articulate things I'd otherwise leave as "intuition." If a junior asks "why fixtures over beforeEach?" — and I can't answer in 30 seconds — I clearly don't know it well enough yet.
6. Use AI as a sparring partner, not a crutch
I use Claude and Cursor heavily — but for thinking with, not thinking instead of. I'll draft a test plan myself, then ask Claude "what risks am I missing?" The combo finds gaps faster than either alone.
